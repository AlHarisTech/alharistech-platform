# ADR-004: استخدام Redis للتخزين المؤقت والجلسات

## الحالة
مقبول (Accepted)

## التاريخ
2026-06-20

## السياق
تحتاج منصة الحارس تك إلى حل للتخزين المؤقت وإدارة الجلسات يدعم:
- **تخزين مؤقت متعدد المستويات** — المنصة صُممت بمعمارية تخزين مؤقت من 4 مستويات (CDN → HTTP Cache → Application Cache → Database Cache). Redis يمثل المستوى الثالث (Application Cache) وهو الأكثر أهمية لتحقيق زمن استجابة API < 200ms (NFR-101).
- **إدارة جلسات المستخدمين** — دعم جلسات متعددة الأجهزة (FR-109) مع إمكانية عرض وإلغاء الجلسات الفردية. تخزين مؤقت لقوائم حظر الرموز (Token Blacklisting) مع TTL.
- **خلفية لنظام الطوابير BullMQ** — BullMQ يعتمد على Redis حصراً كخلفية لتخزين الوظائف وإدارة حالة الطوابير (ADR-009). عمليات الموافقة، معالجة المستندات، إرسال الإشعارات، والأرشفة.
- **تحديد معدل الطلبات (Rate Limiting)** — تنفيذ `@nestjs/throttler` مع نافذة منزلقة (Sliding Window) معتمدة على Redis لدعم معدلات مختلفة: 60 req/min للمجهولين، 300 req/min للعملاء، 1,000 req/min للموظفين (NFR-303 للحماية من هجمات القوة الغاشمة).
- **تخزين مؤقت للاستعلامات** — تخزين نتائج استعلامات قاعدة البيانات المتكررة (قائمة الخدمات، المنتجات، تصنيفات المدونة) مع نمط Cache-Aside و TTL من 5–15 دقيقة.
- **تخزين مؤقت للمتجهات (مرحلة 5)** — تخزين مؤقت لنتائج التضمين (embeddings) لتقليل تكاليف استدعاء OpenAI API وتقليل زمن الاستجابة. نسبة إصابة متوقعة > 60%.
- **أمان** — تشفير TLS 1.3 لاتصال Redis (NFR-301)، حماية من الوصول غير المصرح (NFR-303)، عدم تخزين بيانات حساسة في Redis بدون تشفير (NFR-302).

## محركات القرار
1. الحاجة إلى تخزين مؤقت مركزي سريع جداً (in-memory) لتقليل زمن استجابة API.
2. الحاجة إلى مخزن جلسات موزع يدعم TTL وإلغاء فوري.
3. الاعتماد على BullMQ (ADR-009) الذي يعمل حصراً مع Redis.
4. الحاجة إلى بنية تحتية موحدة — تجنب تشغيل Redis للطوابير + Memcached للتخزين المؤقت.
5. دعم هياكل بيانات متقدمة (Sorted Sets، Hashes، Pub/Sub) لسيناريوهات مثل Rate Limiting، Leaderboards، والتراسل الداخلي.
6. متطلبات التوفر العالي (NFR-201: 99.9% uptime) تستوجب حل Sentinel أو Cluster في الإنتاج.

## الخيارات المدروسة

### الخيار أ: Redis
- **الوصف:** مخزن بيانات In-Memory مفتوح المصدر يدعم هياكل بيانات متعددة (Strings، Hashes، Lists، Sets، Sorted Sets، Streams، Geospatial).
- **إيجابيات:**
  - **أداء فائق** — عمليات قراءة/كتابة في حدود الميكروثانية (Sub-millisecond latency) لأن البيانات في الذاكرة.
  - **متعدد الاستخدامات** — يخدم التخزين المؤقت، الجلسات، Rate Limiting، الطوابير (BullMQ)، Pub/Sub، والتخزين المؤقت للمتجهات في نظام واحد.
  - **هياكل بيانات غنية** — Sorted Sets لتنفيذ Rate Limiting بنافذة منزلقة، Hashes لتخزين الجلسات، Lists للطوابير، Pub/Sub للتراسل بين الخدمات.
  - **TTL مدمج** — انتهاء صلاحية تلقائي للمفاتيح (EXPIRE) يناسب تماماً: جلسات (7 أيام)، أقفال حسابات (15 دقيقة)، قوائم حظر الرموز (حتى انتهاء صلاحية الرمز)، تخزين مؤقت للاستعلامات (5–15 دقيقة).
  - **تكامل أصلي مع BullMQ** — BullMQ لا يدعم أي خلفية أخرى غير Redis، مما يجعل Redis خياراً إلزامياً للطوابير (ADR-009).
  - **Persistenc** — RDB (Snapshot) + AOF (Append Only File) يوفران استمرارية البيانات عبر إعادة التشغيل.
  - **Redis Sentinel & Cluster** — يدعم التوفر العالي (High Availability) مع Sentinel والتوسع الأفقي مع Cluster للمرحلة الخامسة.
  - **أدوات مراقبة ممتازة** — Redis Exporter for Prometheus، RedisInsight للتصحيح والتفتيش.
- **سلبيات:**
  - **نقطة فشل وحيدة (SPOF) في Instance واحد** — إذا تعطل Redis بدون Sentinel، تفقد الجلسات والتخزين المؤقت والطوابير. يجب تخفيفها بـ Sentinel/Cluster.
  - **البيانات في الذاكرة محدودة بحجم RAM** — إذا تجاوزت البيانات الذاكرة المتاحة، يتطلب سياسة إخلاء (Eviction Policy) مناسبة وتخطيطاً دقيقاً لحجم الذاكرة.
  - **Persistenc ليس بديلاً عن النسخ الاحتياطي لقاعدة البيانات** — RDB و AOF ليسا بنفس مستوى ضمانات ACID في PostgreSQL. مناسب للبيانات المؤقتة فقط.
  - **تكلفة ذاكرة أعلى** — مقارنة بقاعدة البيانات، الذاكرة (RAM) أغلى من التخزين (SSD).

### الخيار ب: Memcached
- **الوصف:** مخزن مؤقت In-Memory بسيط، مصمم حصراً للتخزين المؤقت (Key-Value فقط).
- **إيجابيات:**
  - بسيط وسهل الإعداد والتشغيل.
  - أداء ممتاز للتخزين المؤقت البسيط (Key-Value).
  - استهلاك ذاكرة أقل من Redis في بعض الحالات (تجزئة أقل للذاكرة).
- **سلبيات:**
  - **لا يدعم BullMQ** — لا يمكن استخدام Memcached كخلفية لطوابير BullMQ (ADR-009). يجب تشغيل Redis للطوابير + Memcached للتخزين المؤقت = تعقيد إضافي.
  - **لا يدعم هياكل بيانات متقدمة** — لا Sorted Sets (ضرورية لـ Rate Limiting)، لا Pub/Sub (ضرورية للتراسل الداخلي)، لا Streams.
  - **لا يوجد Persistence** — جميع البيانات تُفقد عند إعادة التشغيل. الجلسات والطوابير تحتاج إلى Redis على أي حال.
  - **TTL أقل دقة** — TTL في Memcached يُحسب بالثواني (أقل دقة في السيناريوهات التي تحتاج أجزاء من الثانية).
  - **انتهاء الصلاحية كسول (Lazy Expiration)** — المفاتيح منتهية الصلاحية لا تُحذف فوراً، مما قد يهدر ذاكرة ويُعيد بيانات قديمة.
  - **لا يدعم TLS أصلياً** — يحتاج تشفير الاتصال إلى stunnel أو proxy إضافي (NFR-301).

### الخيار ج: عدم استخدام تخزين مؤقت خارجي (In-Memory Only)
- **الوصف:** الاعتماد على الذاكرة الداخلية للتطبيق (Node.js process memory) للتخزين المؤقت والجلسات.
- **إيجابيات:**
  - لا حاجة لخدمة خارجية — لا Redis، لا Memcached. صفر تعقيد إضافي في البنية التحتية.
  - أسرع استجابة ممكنة (لا شبكة بينية — Network hop).
  - لا تكلفة تشغيل إضافية في المراحل المبكرة.
- **سلبيات:**
  - **لا يعمل مع BullMQ** — BullMQ يتطلب Redis حصراً. بدون Redis، لا يوجد نظام طوابير للموافقات، معالجة المستندات، وإرسال الإشعارات (مخالف لـ ADR-009).
  - **لا يدعم الجلسات الموزعة** — مع 2+ نسخ من NestJS API في الإنتاج (موازنة تحميل)، الجلسة المخزنة في ذاكرة نسخة واحدة غير مرئية للنسخ الأخرى. المستخدم سيفقد جلسته عند توجيهه لنسخة مختلفة (Sticky Sessions ليست حلاً موثوقاً).
  - **لا يدعم Rate Limiting الموزع** — عداد Rate Limiting في ذاكرة نسخة واحدة لا يعكس العدد الإجمالي عبر جميع النسخ.
  - **يفقد البيانات عند إعادة التشغيل** — أي إعادة تشغيل للتطبيق (نشر، تعطل) تؤدي إلى فقدان جميع الجلسات والتخزين المؤقت وعدادات Rate Limiting.
  - **ذاكرة التطبيق محدودة** — Node.js heap محدود (~1.4GB افتراضياً). تخزين كبير في الذاكرة يزيد GC pressure ويؤثر على أداء التطبيق.

## القرار
اخترنا **Redis** للأسباب التالية:

1. **إلزامي لـ BullMQ** — نظام الطوابير BullMQ (ADR-009) يعتمد حصراً على Redis. أي بديل آخر يعني تشغيل Redis للطوابير + حل آخر للتخزين المؤقت، مما يزيد التعقيد دون فائدة.

2. **نظام واحد لخمسة احتياجات** — Redis يخدم: (أ) تخزين مؤقت للاستعلامات، (ب) إدارة الجلسات الموزعة، (ج) Rate Limiting، (د) خلفية طوابير BullMQ، (هـ) تخزين مؤقت للمتجهات (مرحلة 5). تشغيل خدمة واحدة بدلاً من 2–3 خدمات يقلل تكاليف التشغيل والتعقيد.

3. **أداء فائق لتحقيق NFR-101** — Redis في الذاكرة يحقق زمن استجابة < 1ms مقارنة بـ 5–20ms لاستعلام PostgreSQL. هذا ضروري لتحقيق NFR-101 (API p95 < 200ms) مع Pattern Cache-Aside الذي يُجنب قاعدة البيانات في 60–80% من الطلبات المتكررة.

4. **جلسات موزعة أصلية** — مع 2+ نسخ من NestJS API خلف Load Balancer، Redis هو الطبقة الوحيدة المشتركة التي تضمن اتساق الجلسات عبر جميع النسخ دون الحاجة إلى Sticky Sessions.

5. **جاهزية للإنتاج مع Sentinel** — Redis Sentinel يوفر: (أ) كشف تلقائي للفشل (Automatic Failover) خلال ثوانٍ، (ب) إعادة انتخاب Primary تلقائياً، (ج) إشعارات للعملاء (NestJS) بتغيير الهيكل. هذا يحقق NFR-201 (99.9% uptime).

6. **جاهزية للمرحلة الخامسة** — Redis Cluster يوفر Sharding تلقائي لتوزيع البيانات عبر عدة Nodes، مما يدعم التوسع الأفقي عند زيادة الحمل في المرحلة الخامسة (SaaS + Multi-Tenant).

## العواقب

### إيجابيات
- **تخفيض حمل PostgreSQL بنسبة 60–80%** — استعلامات متكررة (قوائم الخدمات، المنتجات، تصنيفات المدونة، إعدادات الموقع) تُخدم من Redis بدلاً من PostgreSQL.
- **Rate Limiting فعّال وموزع** — Sorted Sets في Redis تنفذ نافذة منزلقة بدقة وحجم O(log N)، مما يحمي المنصة من هجمات DDoS والقوة الغاشمة (NFR-303، NFR-305).
- **إدارة جلسات آمنة** — دعم FER-109 (جلسات متعددة الأجهزة): كل جهاز له Refresh Token فريد، يمكن إلغاؤه فوراً عبر `DEL` في Redis. Token Blacklisting مع TTL = المدة المتبقية للرمز.
- **جاهزية لـ Pub/Sub و Event Bus** — في المرحلة الثانية، Redis Pub/Sub يوفر Event Bus بسيطاً للتراسل بين الخدمات (انتقالاً إلى RabbitMQ/Kafka في المرحلة الثالثة).
- **مراقبة شاملة** — Redis Exporter يوفر Metrics تفصيلية (Hit Rate، Memory Usage، Evictions، Connected Clients) إلى Prometheus/Grafana.

### سلبيات
- **نقطة فشل وحيدة (SPOF) في الوضع الأساسي** — Redis Instance واحد بدون Sentinel هو نقطة فشل وحيدة. إذا تعطل: (أ) تفقد جميع الجلسات → جميع المستخدمين يُطرَدون، (ب) تتوقف BullMQ → تتوقف معالجة الإشعارات والطوابير، (ج) ينكسر Rate Limiting → إما يسمح بكل الطلبات أو يرفضها كلها حسب التكوين.
- **تكلفة ذاكرة إضافية** — Redis يحتاج ذاكرة RAM منفصلة (2 vCPU, 4GB RAM في الإنتاج). البيانات المخزنة مؤقتاً + جلسات + طوابير BullMQ يجب ألا تتجاوز 75% من الذاكرة المتاحة.
- **احتمال انفصال البيانات (Cache Invalidation)** — إذا لم تُصمم استراتيجية إلغاء التخزين المؤقت بعناية، قد تعرض المنصة بيانات قديمة (Stale Data). مثلاً: تعديل منتج لكن Redis لا يزال يرسل النسخة القديمة.

### مخاطر
- **خطر: فقدان بيانات الجلسات والطوابير** — تعطل Redis بدون Persistence (RDB/AOF) يعني فقدان جميع الجلسات النشطة (يُطرَد جميع المستخدمين) وجميع الطوابير (تُفقد الوظائف قيد الانتظار). **التخفيف:** (1) تفعيل RDB (Snapshot كل 15 دقيقة) + AOF (كل ثانية) في الإنتاج. (2) Redis Sentinel لاكتشاف الفشل والتحويل التلقائي إلى Replica. (3) الطوابير الحرجة (مثل معالجة الدفع) تُصمم مع Idempotency Keys لإعادة المحاولة بأمان. (4) الجلسات يمكن إعادة إنشائها (المستخدم يسجل الدخول مجدداً) — لا تُفقد بيانات دائمة.
- **خطر: تجاوز الذاكرة (OOM)** — إذا تجاوزت البيانات الذاكرة المتاحة، قد يقتل نظام التشغيل Redis (OOM Killer) أو يبدأ Redis في رفض الكتابة (إذا وصل إلى `maxmemory`). **التخفيف:** (1) إعداد `maxmemory-policy allkeys-lru` لإخلاء المفاتيح الأقل استخداماً تلقائياً. (2) تحديد `maxmemory` بنسبة 75% من RAM المتاحة. (3) مراقبة `redis_memory_used_bytes` وإطلاق إنذار عند > 80%. (4) تقدير حجم الذاكرة المطلوبة: ~1GB للجلسات + ~1GB للتخزين المؤقت + ~500MB للطوابير = 2.5GB (مع هامش أمان 4GB).
- **خطر: هجوم على Redis** — Redis لا يدعم Authentication افتراضياً. إذا تُرك مفتوحاً على الإنترنت بدون حماية، يمكن اختراقه بسهولة. **التخفيف:** (1) `requirepass` كلمة مرور قوية. (2) Redis لا يُعرض على الإنترنت مطلقاً — فقط داخل شبكة Docker الداخلية. (3) تعطيل الأوامر الخطيرة في الإنتاج: `FLUSHALL`، `FLUSHDB`، `CONFIG`، `DEBUG`، `SHUTDOWN` عبر `rename-command` أو `ACL`. (4) تشفير TLS 1.3 لاتصال Redis (NFR-301). (5) تقييد الاتصال بـ `bind` إلى واجهة الشبكة الداخلية فقط.

## الامتثال
- **فحص تلقائي:** GitHub Actions يتحقق من تكوين `docker-compose.yml` للإنتاج: `requirepass` موجود، `rename-command` للأوامر الخطيرة، `maxmemory-policy` محدد.
- **مراجعة:** تغييرات تكوين Redis (خاصة في `redis.conf` للإنتاج) تتطلب مراجعة من DevOps/Lead Developer.
- **مراقبة:** إنذار فوري (Pager) عند: `redis_memory_used_bytes > 80%`، `redis_connected_clients > 500`، `redis_evicted_keys_total > 0 (زيادة)`.
- **اختبار اختراق دوري:** OWASP ZAP (أسبوعي) يتحقق من عدم تعرض Redis على الإنترنت (NFR-303).
- **تشفير:** TLS 1.3 مفروض للاتصال بين NestJS و Redis في الإنتاج. كلمة مرور Redis تُمرر عبر متغيرات البيئة (NFR-301، NFR-302).

## القرارات ذات الصلة
- ADR-003: PostgreSQL كقاعدة بيانات أساسية — Redis يُكمل PostgreSQL كطبقة تخزين مؤقت لتقليل الضغط.
- ADR-007: JWT + Refresh Token — Redis يُستخدم لتخزين قوائم حظر الرموز (Token Blacklist) وإدارة الجلسات.
- ADR-009: BullMQ للطوابير — BullMQ يعتمد على Redis حصراً كخلفية.
- ADR-006: Architecture Model — Redis هو Adapter في الطبقة التحتية (Infrastructure Layer) ضمن البنية السداسية.
- ADR-011: Docker — Redis يُشغل كحاوية Docker في جميع البيئات (dev، staging، production).
