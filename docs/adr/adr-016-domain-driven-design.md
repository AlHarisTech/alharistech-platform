# ADR-016: اعتماد Domain-Driven Design (DDD) Tactical Patterns

## الحالة
مقبول (Accepted)

## السياق
المنصة تحتوي على 9 نطاقات محددة (bounded contexts) مع قواعد عمل معقدة (مثل حسابات العملاء، إدارة المخزون، نظام الطلبات). المشروع سيتطور عبر 6 مراحل على مدى سنوات، مما يتطلب هيكلة تسمح بتطور مستقل لكل نطاق وتجنب تشابك المنطق (spaghetti business logic).

## الخيارات المدروسة

### الخيار أ: Full DDD Tactical Patterns
- إيجابيات: Aggregates تحمي invariants؛ Domain Events تمكن التواصل غير المتزامن بين النطاقات؛ Repository pattern يعزل منطق الاستعلام؛ Value Objects تغلف منطق القيم؛ فصل واضح بين النطاقات يسمح بتطورها المستقل
- سلبيات: منحنى تعلم مرتفع للفريق؛ انضباط عالٍ مطلوب للالتزام بالأنماط؛ وقت تطوير أطول في البداية

### الخيار ب: Anemic Domain Model (نماذج بدون منطق)
- إيجابيات: بسيط وسريع في البداية؛ مألوف لمعظم المطورين
- سلبيات: منطق العمل يتوزع في services مما يؤدي إلى تشابك؛ صعوبة في ضمان invariants؛ لا تطور مستقل للنطاقات؛ يصبح غير قابل للصيانة مع نمو التعقيد

### الخيار ج: Transaction Script (إجراءات بسيطة)
- إيجابيات: الأسرع في التنفيذ؛ مناسب للتطبيقات البسيطة
- سلبيات: لا يناسب تعقيد المنصة؛ تكرار المنطق بين العمليات المختلفة؛ صعوبة في الاختبار والصيانة مع نمو النظام

## القرار
اخترنا **Full DDD Tactical Patterns** للأسباب التالية:
1. **Tactical Patterns**: استخدام Aggregates مع invariants، Domain Events للتواصل، Repository pattern لعزل استعلامات البيانات، Value Objects للقيم ذات المنطق
2. **Strategic DDD**: كل Bounded Context يطابق مجلد `domains/` في الـ monorepo (مثلاً `domains/customers/`, `domains/inventory/`, `domains/orders/`)
3. **Event-Driven Communication**: Domain Events تنشر عبر EventBus داخلي (NestJS EventEmitter) في المرحلة الحالية
4. **تأجيل CQRS**: CQRS الكامل (مع Event Sourcing) يؤجل للمرحلة الثانية وما بعدها — حالياً نستخدم Read Models بسيطة
5. **Ubiquitous Language**: كل domain يستخدم لغة موحدة (Ubiquitous Language) مشتقة من قاموس الأعمال

## العواقب
- إيجابيات: حدود واضحة بين النطاقات تسمح بتطور مستقل؛ منطق عمل مغلف وقابل للاختبار؛ سهولة إضافة نطاقات جديدة؛ دعم التطور عبر المراحل دون إعادة كتابة
- سلبيات: انضباط عالٍ مطلوب من جميع المطورين (code review إلزامي للتحقق من الالتزام بالأنماط)؛ وقت تطوير أطول في البداية (~20% overhead)
- مخاطر: الإفراط في الهندسة (over-engineering) في المراحل المبكرة — نخففها بالبدء بـ DDD tactical فقط دون CQRS/Event Sourcing في المرحلة الأولى؛ مقاومة الفريق للانضباط — نخففها بالتدريب ومراجعات الكود المنتظمة
