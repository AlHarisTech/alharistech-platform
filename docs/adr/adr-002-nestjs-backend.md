# ADR-002: استخدام NestJS للخدمات الخلفية

## الحالة
مقبول (Accepted)

## السياق
نحتاج إلى إطار عمل للخادم (Backend) يدعم:
- المعمارية المؤسسية (Enterprise Architecture)
- النمطية (Modularity)
- Dependency Injection
- OpenAPI/Swagger تلقائي
- GraphQL جاهزية
- WebSockets
- Queues

## الخيارات المدروسة

### الخيار أ: NestJS
- إيجابيات: معماري مؤسسي، DI مدمج، OpenAPI تلقائي، GraphQL + WS + Queues، يشبه Angular في الهيكلة، TypeScript أصلي
- سلبيات: منحنى تعلم، boilerplate أكثر

### الخيار ب: Express.js
- إيجابيات: بسيط، مجتمع ضخم، مرن
- سلبيات: لا هيكلة مفروضة، يحتاج بناء كل شيء يدوياً

### الخيار ج: Fastify
- إيجابيات: أداء عالي جداً، TypeScript
- سلبيات: مجتمع أصغر، نظام إضافات أقل

## القرار
اخترنا **NestJS** للأسباب التالية:
1. المعمارية المؤسسية تناسب حجم وتعقيد المنصة
2. النمطية (Modules) تسهل فصل النطاقات (Domains)
3. DI مدمج يسهل الاختبار والصيانة
4. OpenAPI/Swagger تلقائي يدعم API First
5. دعم GraphQL و WebSockets و Queues جاهز

## العواقب
- إيجابيات: هيكلة واضحة، نمطية، سهولة التوسع
- سلبيات: منحنى تعلم أعلى للمطورين الجدد
- مخاطر: الإفراط في الهندسة (over-engineering) للمراحل الأولى — نخففها بالبدء ببنية بسيطة نسبياً
