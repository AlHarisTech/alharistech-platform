# ADR-003: استخدام PostgreSQL كقاعدة بيانات أساسية

## الحالة
مقبول (Accepted)

## التاريخ
2026-06-20

## السياق
تحتاج منصة الحارس تك إلى قاعدة بيانات أساسية تدعم:
- **معاملات ACID** لضمان سلامة البيانات عبر نطاقات متعددة (الطلبات، الفواتير، التذاكر، المدفوعات).
- **بيانات علائقية معقدة** — المنصة تحتوي على 9 نطاقات محددة (Bounded Contexts) مع علاقات متشابكة بين المستخدمين، العملاء، الطلبات، المنتجات، التذاكر، الفواتير، والإشعارات.
- **استعلامات تحليلية** للوحة التحكم والتقارير (FR-901 إلى FR-906) مع الحاجة إلى Materialized Views واستعلامات مجمّعة.
- **بحث نصي** للمقالات والخدمات والمنتجات باللغتين العربية والإنجليزية (نطاق المحتوى والمتجر).
- **جاهزية للتوسع** — دعم 100,000+ طلب خدمة (NFR-402) و50,000+ منتج (NFR-403) و10,000+ مستخدم مسجل (NFR-401) مع إمكانية التقسيم (Partitioning) في المرحلة الخامسة.
- **قيود أمنية** — تشفير البيانات المخزنة (NFR-302/AES-256)، تشفير الحقول الحساسة (الرقم القومي، الرقم الضريبي)، سجل تدقيق شامل (Audit Logging).
- **حماية OWASP Top 10** (NFR-303) على مستوى قاعدة البيانات (SQL Injection، إدارة الصلاحيات الدقيقة).

## محركات القرار
1. الحاجة إلى معاملات ACID موثوقة عبر النطاقات (خاصة المدفوعات والفواتير والطلبات).
2. الحاجة إلى استعلامات علائقية معقدة (JOIN، GROUP BY، نافذة) للتقارير ولوحات التحكم.
3. دعم البحث النصي بالعربية والإنجليزية مع فهرسة متقدمة (GIN، GIST، pg_trgm).
4. مرونة في التعامل مع بيانات شبه مهيكلة (JSONB) لحقول مثل `dimensions_cm` و `changes` في سجل التدقيق.
5. جاهزية للمتجهات (pgvector) لخط أنابيب RAG في المرحلة الخامسة.
6. دعم التقسيم (Partitioning) للجداول الضخمة (>10M صف) في المراحل المتقدمة.

## الخيارات المدروسة

### الخيار أ: PostgreSQL
- **الوصف:** قاعدة بيانات علائقية مفتوحة المصدر تدعم ACID، JSONB، البحث النصي، التقسيم، وامتدادات متخصصة.
- **إيجابيات:**
  - ACID كامل مع مستويات عزل متعددة (Read Committed، Repeatable Read، Serializable).
  - دعم JSONB يسمح بمزاياه البيانات العلائقية مع مرونة البيانات شبه المهيكلة دون التضحية بالفهرسة والاستعلامات المتقدمة.
  - بحث نصي متكامل (Full-Text Search) مع دعم اللغة العربية عبر `arabic_stem` وقواميس مخصصة.
  - امتداد `pgvector` جاهز لتخزين المتجهات والاستعلام عنها (RAG Pipeline في المرحلة الخامسة).
  - امتداد `pg_trgm` للبحث التقريبي (Fuzzy Search) والتشابه النصي.
  - تقسيم جدولي (Table Partitioning) مدمج منذ الإصدار 14 (Range، List، Hash).
  - Materialized Views للاستعلامات التحليلية المُجمّعة مسبقاً.
  - PgBouncer لتجميع الاتصالات (Connection Pooling) في الإنتاج.
  - فهرسة متقدمة: B-tree، GIN، GiST، BRIN، Partial، Covering.
  - مجتمع ضخم، توثيق ممتاز، دعم طويل الأمد، ترخيص PostgreSQL.
- **سلبيات:**
  - قابلية التوسع الأفقي محدودة مقارنة بقواعد NoSQL (لكن التقسيم + النسخ المتماثلة (Read Replicas) تغطي الاحتياجات حتى المرحلة الخامسة).
  - إعداد التقسيم يحتاج تخطيطاً إضافياً (لا يتم تلقائياً).
  - البحث النصي أقل قوة من Elasticsearch (لكنه كافٍ للمراحل 1–3؛ Elasticsearch يُعتمد في المرحلة 4+ للبحث المتقدم).
  - أداء JSONB أقل من قواعد المستندات الأصلية في الاستعلامات المعقدة على مستندات كبيرة جداً (لكنه كافٍ لحالات الاستخدام الحالية).

### الخيار ب: MySQL
- **الوصف:** قاعدة بيانات علائقية مفتوحة المصدر واسعة الانتشار، تدعم ACID عبر InnoDB.
- **إيجابيات:**
  - انتشار واسع وسهولة في إيجاد خبراء ومطورين.
  - أداء قراءة ممتاز مع InnoDB.
  - دعم جيد للتكرار (Replication) والتقسيم.
  - أدوات إدارة متعددة (MySQL Workbench، phpMyAdmin).
- **سلبيات:**
  - JSON أقل نضجاً من PostgreSQL (يفتقر إلى الفهرسة المتقدمة على JSON مثل GIN على JSONB).
  - البحث النصي أقل دعماً للعربية — لا يدعم قواميس عربية مخصصة بسهولة، ويقتصر على ngram parser.
  - لا يدعم `pgvector` ولا Materialized Views بشكل أصلي (يحتاج بدائل أقل كفاءة).
  - لا يدعم BRIN indexes ولا Partial indexes المتقدمة.
  - دعم محدود للامتدادات (Extension ecosystem أقل بكثير من PostgreSQL).
  - إعداد التقسيم أقل مرونة (لا يدعم التقسيم بقيم افتراضية قبل الإصدار 8.0).

### الخيار ج: MongoDB
- **الوصف:** قاعدة بيانات مستندات NoSQL بدون مخطط (Schema-less).
- **إيجابيات:**
  - مرونة عالية في هيكل البيانات — لا حاجة لتعريف المخطط مسبقاً.
  - قابلية توسع أفقي ممتازة عبر Sharding.
  - أداء قراءة/كتابة عالي في بعض السيناريوهات.
- **سلبيات:**
  - **غياب ACID الكامل عبر المستندات المتعددة** — يدعم ACID على مستوى المستند الواحد فقط. العمليات عبر النطاقات (طلب → فاتورة → دفع) قد تفقد Atomicity بدون تعقيد إضافي (Two-Phase Commit يدوي).
  - العلاقات بين المجموعات (Joins) ضعيفة الأداء (`$lookup` أبطأ بكثير من JOIN في PostgreSQL).
  - لا يدعم البحث النصي العربي بقواميس مخصصة — يعتمد على Lucene الأساسي.
  - لا يوجد مفهوم Materialized Views أو VIEWS أصلي.
  - التقارير والاستعلامات التحليلية المعقدة (NFR-902–NFR-906) تتطلب Aggregation Pipeline معقد وأداء أقل من SQL.
  - لا يدعم `pgvector` (يحتاج Atlas Vector Search المدفوع).
  - معظم بيانات المنصة **علائقية بطبيعتها**: مستخدم ← عميل ← طلب ← فاتورة ← دفعة؛ تذكرة ← عميل ← طلب. نمذجة هذه العلاقات في MongoDB تؤدي إلى تضخم المستندات (Embedding) أو استعلامات `$lookup` بطيئة.

## القرار
اخترنا **PostgreSQL** للأسباب التالية:

1. **البيانات علائقية بامتياز** — المنصة تحتوي على 9 نطاقات محددة مع علاقات Foreign Key معقدة (مستخدم ← عميل ← طلب ← فاتورة ← دفعة، تذكرة ← موظف، منتج ← مخزون). PostgreSQL يتفوق في هذا النوع من العلاقات مع JOINs فعّالة وتكامل مرجعي (Foreign Key Constraints).

2. **مرونة JSONB للبيانات شبه المهيكلة** — حقول مثل `dimensions_cm` في المنتجات، `changes` في سجل التدقيق، وإعدادات المستخدم تُخزن بمرونة في JSONB مع إمكانية الفهرسة والاستعلام المتقدم (عكس MySQL JSON المحدود).

3. **جاهزية للذكاء الاصطناعي** — امتداد `pgvector` يوفر قاعدة بيانات متجهات أصلية لخط أنابيب RAG دون الحاجة لخدمة منفصلة (Pinecone/Weaviate)، مما يبسط البنية ويقلل التكاليف التشغيلية.

4. **بحث نصي عربي** — PostgreSQL هو الخيار الوحيد الذي يدعم قواميس بحث عربية مخصصة (`arabic_stem`) مع فهرسة GIN متقدمة. هذا ضروري للمدونة (FR-404) والمنتجات (FR-701) وصفحات المحتوى (FR-401).

5. **Materialized Views للتقارير** — يدعم PostgreSQL إنشاء Materialized Views (`mv_daily_order_stats`، `mv_customer_ltv`، `mv_ticket_sla_performance`) تُجدول تحديثها دورياً، مما يفي بمتطلبات التقارير والتحليلات (FR-901–FR-906) بأداء عالٍ.

6. **استراتيجية تقسيم للمرحلة الخامسة** — يدعم PostgreSQL التقسيم Range partitioning على أعمدة `created_at` للجداول الضخمة (الطلبات، الفواتير، التذاكر، الإشعارات، سجل التدقيق) عند تجاوز ~10 مليون صف، مما يضمن أداءً مستداماً مع نمو البيانات.

7. **توافق مع NFR-303 (OWASP)** — PostgreSQL يدعم تشفير TDE، صلاحيات دقيقة على مستوى الصفوف (Row-Level Security) للمرحلة الخامسة (Multi-Tenant)، وصلاحيات على مستوى الأعمدة.

## العواقب

### إيجابيات
- **سلامة بيانات كاملة** — قيود Foreign Key، Unique، Check، و ACID تضمن عدم فقدان أو تلف البيانات، خاصة في العمليات المالية (فواتير، مدفوعات).
- **استعلامات تحليلية قوية** — Materialized Views و Window Functions و CTEs تمكن تقارير معقدة دون الحاجة إلى ETL منفصل في المراحل المبكرة.
- **قاعدة بيانات واحدة لمتطلبات متعددة** — PostgreSQL يخدم احتياجات علائقية (كل النطاقات)، بحث نصي (المحتوى، المتجر)، متجهات (الذكاء الاصطناعي)، وJSONB (بيانات مرنة) في نظام واحد.
- **تكاليف تشغيلية أقل** — لا حاجة لتشغيل Elasticsearch أو Pinecone في المراحل المبكرة. PostgreSQL يغطي البحث النصي حتى المرحلة الثالثة.
- **امتدادات جاهزة للمستقبل** — `pgvector` (مرحلة 5)، `postgis` (إذا أضيفت ميزات مواقع)، `pg_cron` (للوظائف المجدولة داخل قاعدة البيانات).

### سلبيات
- **قابلية التوسع الأفقي محدودة** — PostgreSQL لا يدعم التوزيع التلقائي (Sharding). الحل: Read Replicas + Partitioning يغطيان الاحتياجات حتى مرحلة متقدمة جداً. عند الحاجة لـ Sharding، يمكن الانتقال إلى Citus (PostgreSQL extension).
- **البحث النصي أقل قوة من Elasticsearch** — لا يدعم faceted search، autocomplete المتقدم، أو typo tolerance خارج الصندوق. الحل: Elasticsearch يُضاف في المرحلة 4+ للبحث المتقدم. PostgreSQL FTS + pg_trgm كافٍ للمراحل 1–3.
- **إدارة التقسيم يدوية** — لا يوجد تقسيم تلقائي للجداول. الحل: سياسة تقسيم محددة مسبقاً (شهري) + أتمتة عبر Cron jobs أو BullMQ.

### مخاطر
- **خطر: اختناق قاعدة بيانات واحد** — جميع النطاقات تعتمد على نفس PostgreSQL instance في المراحل المبكرة. **التخفيف:** (1) PgBouncer لتجميع الاتصالات ومنع استنزافها. (2) Read Replicas لفصل استعلامات القراءة (التقارير، لوحات التحكم) عن الكتابة. (3) Redis للتخزين المؤقت (ADR-004) يقلل الضغط على PostgreSQL. (4) خطة للانتقال إلى CQRS + Event Sourcing في المرحلة الثالثة لفصل القراءة عن الكتابة بشكل كامل.
- **خطر: أداء JSONB مع مستندات كبيرة جداً** — استعلامات JSONB المعقدة قد تكون أبطأ من MongoDB. **التخفيف:** (1) تحديد حجم حقول JSONB (لا تتجاوز بضع KB). (2) فهرسة GIN على مسارات JSONB المستخدمة في الاستعلامات. (3) نقل البيانات شبه المهيكلة كبيرة الحجم إلى S3/ MinIO إذا تجاوزت الحدود.
- **خطر: نمو حجم قاعدة البيانات** — مع 100,000+ طلب و50,000+ منتج، قد يصل حجم البيانات إلى مئات GB. **التخفيف:** (1) سياسة تقسيم Monthly partitioning على الجداول الكبيرة. (2) أرشفة البيانات القديمة (Cold Storage → S3 Glacier) بعد فترة الاحتفاظ المحددة (7 سنوات للطلبات، 3 سنوات للتذاكر). (3) تنظيف دوري للجداول المؤقتة (سجل الإشعارات، سجل الجلسات).

## الامتثال
- **فحص تلقائي:** GitHub Actions يتحقق من صحة ملفات `schema.prisma` (أو Drizzle Kit migrations) عند كل PR.
- **مراجعة:** كل تغيير في مخطط قاعدة البيانات يتطلب مراجعة من مهندس أساسي (Lead Developer).
- **سياسة Expand-Contract:** جميع تغييرات المخطط يجب أن تكون متوافقة مع الإصدارات السابقة (Backward Compatible). لا يُسمح بحذف الأعمدة أو الجداول مباشرة — تُهمل أولاً ثم تُحذف بعد دورة هجرة كاملة.
- **أمان:** كلمات مرور قاعدة البيانات لا تُخزن في الكود المصدري — تُمرر عبر متغيرات البيئة (`.env`) أو Docker Secrets. تشفير TLS 1.3 للاتصال بين التطبيق وقاعدة البيانات (NFR-301).

## القرارات ذات الصلة
- ADR-004: Redis للتخزين المؤقت والجلسات — يكمل PostgreSQL بتقليل ضغط الاستعلامات المتكررة.
- ADR-006: Modular Layered + DDD + Hexagonal Architecture — نمط Repository في البنية السداسية يفصل PostgreSQL عن منطق النطاق.
- ADR-010: S3-Compatible Storage — الملفات والوسائط تُخزن في S3/MinIO وليس في PostgreSQL.
- ADR-016: Domain-Driven Design (DDD) — كل نطاق له جداوله الخاصة ضمن نفس قاعدة البيانات (Database per Service في المرحلة الخامسة إذا لزم).
