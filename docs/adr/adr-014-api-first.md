# ADR-014: اعتماد API First Design

## الحالة
مقبول (Accepted)

## السياق
المنصة ستخدم عبر قنوات متعددة (web, desktop, mobile) تستهلك جميعها نفس API. نحتاج إلى عقود API متسقة ومضمونة التوافق بين جميع المستهلكين منذ البداية، مع تجنب الانجراف (drift) بين التوثيق والتنفيذ.

## الخيارات المدروسة

### الخيار أ: API First — كتابة OpenAPI قبل الكود
- إيجابيات: عقد واضح ومتفق عليه قبل التنفيذ؛ جميع الفرق (frontend, mobile, desktop) تبدأ بالتوازي بمجرد توفر الـ spec؛ التوثيق هو المصدر الوحيد للحقيقة (single source of truth)؛ يمنع التضارب بين القنوات
- سلبيات: وقت إضافي في البداية لكتابة وصيانة الـ spec يدوياً؛ يحتاج انضباط لتحديث الـ spec مع كل تغيير

### الخيار ب: Code First — توليد التوثيق من الكود
- إيجابيات: أسرع في البداية؛ التوثيق يتولد تلقائياً من الـ decorators؛ لا حاجة لكتابة وصيانة spec منفصل
- سلبيات: الكود يصبح المصدر الوحيد للحقيقة — الفرق الأخرى تنتظر اكتمال الـ endpoint؛ لا يوجد عقد مسبق للتفاوض عليه؛ قد يؤدي إلى APIs غير متسقة

### الخيار ج: Hybrid — كتابة الخطوط العريضة يدوياً وتفاصيل النماذج من الكود
- إيجابيات: مرونة في اختيار ما يكتب يدوياً وما يتولد تلقائياً
- سلبيات: مصدران للحقيقة يؤديان إلى تناقضات؛ تعقيد إضافي في سير العمل

## القرار
اخترنا **API First مع OpenAPI 3.1** للأسباب التالية:
1. كتابة مواصفة OpenAPI 3.1 قبل بدء التنفيذ لكل domain
2. NestJS يقوم بتوليد OpenAPI spec تلقائياً من الـ decorators للتحقق من التطابق (validation)
3. تخزين العقود النهائية في `specs/contracts/` كمصدر وحيد للحقيقة
4. استخدام أدوات مثل Spectral لـ linting الـ OpenAPI spec
5. توليد TypeScript types من الـ spec باستخدام openapi-typescript لضمان تطابق types الواجهة الأمامية

## العواقب
- إيجابيات: عقود API متسقة لكل القنوات؛ توثيق دائم التزامن؛ الفرق تبدأ بالتوازي؛ تقليل أخطاء التكامل
- سلبيات: وقت إضافي في المرحلة الأولى لكتابة الـ spec (يقدر بـ 10-15% من وقت التطوير)
- مخاطر: إهمال تحديث الـ spec اليدوي مع الوقت — نخففها باستخدام CI check يقارن الـ auto-generated spec مع الـ stored spec ويفشل الـ build عند وجود اختلاف
